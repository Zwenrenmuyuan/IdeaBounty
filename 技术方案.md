# Idea Bounty 技术方案

> 状态：后端投稿、评估、查重、红包估值和管理员模拟打款闭环已实现；前端待实现。
> 更新日期：2026-07-11

## 1. 项目目标

Idea Bounty 是一个商业点子收集器。注册用户提交生活痛点或商业点子后，系统通过 AI 评估商业价值、判断是否与历史点子语义重复，并计算虚拟红包金额。每个点子和金额都归属于明确用户，用户可以查看自己的提交历史。管理员可以查看、复核、调整或驳回评估结果，并在所有前置流程完成后确认一笔不调用真实支付渠道的模拟打款。

本项目是 8–16 小时有效开发时间内的 MVP，优先保证核心闭环、可演示性和可测试性，不追求完整的商业化功能。

## 3. 总体架构

```text
浏览器
├── 用户前端
└── 管理后台
        │
        ▼
React + Vite + TypeScript + Tailwind
        │ REST/JSON
        ▼
FastAPI
├── 用户注册、登录与会话
├── 点子提交与查询
├── AI 评估流程
├── Embedding 候选召回 + LLM 查重判定
├── 红包规则计算
├── 管理员复核
└── 模拟打款确认
        │
        ▼
PostgreSQL
```

前后端通过 REST API 交互。后端负责所有业务规则，前端不直接决定 AI 评分、重复折扣、最终红包金额或模拟打款状态。

## 4. AI 服务与输出约定

AI 实现不绑定某个服务商或模型。后端通过环境变量接收与 OpenAI Chat Completions 兼容的配置：

- `AI_BASE_URL`
- `AI_API_KEY`
- `AI_MODEL_ID`
- `AI_TIMEOUT_SECONDS`（可选，默认 60 秒）
- `AI_MAX_RETRIES`（可选，默认追加 2 次，范围 0–3）
- `AI_TEMPERATURE`（可选，默认 0.2，范围 0–2）

生成模型请求使用服务端实际支持的 `response_format.type = json_object`。该模式只保证返回合法 JSON，不保证字段、层级和业务约束。后端把 Pydantic 生成的完整 JSON Schema 压缩后放入系统提示词，明确要求模型返回全部字段、禁止额外字段并遵守类型、枚举和 `null` 规则；用户原文仍作为单独的不可信数据传入，不与系统指令拼接。

模型响应必须先解析为 JSON，再通过 Pydantic 和 Python 跨字段校验。提示词中的 Schema 用于提高模型遵守率，Pydantic 才是写入数据库前的强制边界。当前 `evaluation-v2` 将 `unknown/value` 组合、证据字段枚举、有无明确方案和门禁结果联动直接编码进 JSON Schema，并在 Prompt 中重复最容易违反的边界。后端不删除未知字段、不补造缺失字段、不把错误类型静默转换，也不从校验失败的结果中部分提取数据。

需要区分并记录以下错误：

1. **JSON mode 不支持**：服务端以 400/422 等状态拒绝 `json_object` 参数。
2. **响应格式错误**：响应不符合 Chat Completions 格式、内容为空、不是合法 JSON 或因长度限制被截断。
3. **业务契约未遵守**：JSON 可以解析，但缺少字段、包含额外字段、类型或枚举错误，或无法通过跨字段校验。
4. **鉴权或配置错误**：Base URL、API Key 或 Model ID 无效。
5. **运行时失败**：超时、限流、拒答或服务商内部错误。

上述错误都不得产生规范化结果、评分或金额，也不得部分写入 JSONB。处理记录进入 `processing_status=failed`，保存安全的 `failure_stage/failure_code`；后续重试重新发起完整请求，不对失败输出做宽松修补。

自动重试只用于连接错误、超时、429、5xx、空或截断输出、非法 JSON 以及 Pydantic 校验失败；400/401/403/404/422 等配置、鉴权或能力错误立即失败。追加重试使用 `1s → 2s → 4s` 指数退避并加入最多 250ms 抖动。自动重试不增加 `ideas.retry_count`，该字段只统计用户在一整轮自动重试耗尽后主动发起的阶段重试。

前端对普通用户只显示“AI 评估暂时不可用”和请求编号，不泄露服务商错误细节。后端日志只记录经过 ASCII 字段名过滤的 `字段路径:错误类型` 签名，不记录用户原文、模型字段值、完整输出或 API Key。

后端将使用 Pydantic 对 AI 结果进行强制后处理校验，包括：

- 所有必填评分维度是否存在。
- 评分是否在规定区间内。
- 查重结果是否为 `duplicate | related | novel` 中的合法枚举值。
- 重复时是否包含所需的匹配信息。
- 理由和摘要是否符合长度限制。
- 是否包含未定义字段或错误数据类型。

LLM 主要负责分析和评分，最终红包金额由后端使用明确的 Python 规则计算，以提高稳定性、可解释性和可测试性。

### 4.1 模型服务能力探测

实现阶段提供一个独立的 Python 探测脚本。当前脚本从 AI 环境变量读取配置，将实际 `EvaluationOutput` Schema 放入系统提示词，并用包含真实痛点、自夸断言、提示词注入和红包指令的固定样例测试生成模型，而不只验证一个简单 JSON 对象。脚本输出：

- Chat Completions 兼容端点是否可访问。
- 鉴权和 Model ID 是否有效。
- 是否接受 `response_format.type=json_object`。
- 返回内容是否为合法 JSON 并通过实际 `EvaluationOutput` 的 Pydantic 与跨字段校验。
- 是否保留有效投稿，同时识别提示词注入、指定金额和未证实市场断言。
- 每次请求耗时、服务端返回的 token 用量和可得的请求 ID。

任何 JSON 解析、Pydantic 校验或最低行为检查失败时，脚本使用非零退出码，不降级到普通文本，也不把宽松解析当作通过。Embedding 和查重判定分别使用独立探测脚本，避免把不同模型能力混在一个结果中。

2026 年 7 月的首次实际探测中，`mimo-v2.5-pro` 能通过 `json_object` 和完整
`EvaluationOutput` 后处理校验，但固定抗诱导样例只识别了指定金额，没有同时识别“忽略前面的规则”这一提示词注入。补充五类诱导信号的明确边界、多标签累计规则和固定示例后，第二次探测通过全部检查：有效投稿被保留，提示词注入、指定红包金额和未证实市场断言均被识别。相同服务对 `json_schema + strict` 没有强制执行传入结构。MiMo 官方文档也明确其 Structured Output 是只保证合法 JSON 的 `json_object`，字段结构需要通过提示词描述并在客户端校验：[MiMo Structured Output](https://mimo.mi.com/docs/en-US/quick-start/usage-guide/text-generation/structured-output)。该结果确认当前接入方案可用，但也表明抗诱导表现会受提示词约束影响；单次通过不代表长期稳定，后续仍需用多样例和重复运行评估结构通过率、延迟和评分波动。

生产联调进一步发现原契约和 `evaluation-v2` 初版仍会偶发输出 `unknown` 与非空值并存、证据字段越过白名单、`accept` 同时携带补充问题，以及把期望结果扩写成方案。`evaluation-v2` 初版在未显式控制随机性的 20 次对照中取得 14/20 契约与行为通过：4 次违反 `accept/clarification_question` 联动，2 次输出证据枚举外值。随后补充门禁决策的 JSON Schema 联动、将证据引用限制为 1–3 个枚举，并显式使用 `temperature=0.2`；同样的“无明确方案”和“明确方案 + 提示词注入”交替样例共 20 次全部通过，且方案判定分别稳定为 `false/true`。MiMo 官方说明 `mimo-v2.5-pro` 默认温度为 1.0，低温度会降低随机性：[MiMo 模型超参数](https://mimo.mi.com/docs/en-US/api/guidance/model-hyperparameters)。20/20 只证明当前小样例明显改善，不代表所有真实投稿都不会触发重试，后续仍需扩大评测集。

### 4.2 模型选择原则

不根据模型名称或服务商宣传直接锁定模型，而是选择“能通过项目评测集的最低成本模型”。评测指标包括：

- JSON 合法率和 Pydantic 业务契约通过率。
- 对提示词注入和自我夸大语句的抵抗能力。
- 高价值和低价值样本的排序正确率。
- 相同输入多次评估的稳定性。
- 中文理由的清晰度。
- 单次调用成本和延迟。

评测集使用“预期档位和相对排序”作为标注，不强求主观的精确分数完全一致。

## 5. 用户输入处理与抗诱导

### 5.1 最小化提交表单

用户只需填写一个必填文本字段，不要求用户手动填写标题、分类、目标用户和结构化评估维度。

表单文案：

> 说说你遇到的问题或想到的解决办法。尽量说明谁在什么场景遇到了什么麻烦；如果有解决办法，也可以一起写。

输入长度限制为 8–2000 个字符。字符数只负责拦截空内容和极端输入，真正的信息质量由输入门禁判断。原始文本完整保存，同时保存 LLM 生成的规范化结果，管理员可以对照查看。

### 5.2 规范化提取

痛点和商业点子共用一组基础字段：

- `target_audience`：面向谁。
- `pain_point`：核心痛点。
- `context`：发生场景。
- `frequency_or_severity`：频率或严重程度。
- `current_alternative`：当前如何应对。
- `desired_outcome`：期望结果。

如果用户提供了解决方案，再额外提取：

- `proposed_solution`：用户提出的方案。
- `solution_mechanism`：方案如何起作用。
- `value_proposition`：相对当前做法的改进。

LLM 可以生成简短标题、改写口语表达、提取结构和谨慎推断上下文，但不得编造市场规模、调研数据、付费意愿或用户没有提出的解决方案。

每个关键字段同时记录信息来源：

- `explicit`：用户在原文中明确表达。
- `inferred`：可以从上下文合理推断。
- `unknown`：原文没有足够信息。

`inferred` 不作为与 `explicit` 同等强度的商业价值证据，`unknown` 不得被自动补造。只有用户明确提供了方案时，`solution_present` 才能为真。

### 5.3 输入门禁

在评分和查重前，LLM 必须先输出以下三种决策之一：

| 决策 | 适用情况 | 系统行为 |
| --- | --- | --- |
| `accept` | 可以识别明确痛点或用户提出的方案 | 继续商业价值评估和查重 |
| `clarify` | 看起来是真实投稿，但过于模糊 | 给出一个聚焦的补充问题，暂不计算红包 |
| `reject` | 纯提示词注入、垃圾内容、重复字符或完全无关 | 明确拒绝，不调用后续估值和查重 |

处理优先级是：

```text
能提取真实贡献 -> 清洗诱导表达后接受
可能有真实贡献但不清楚 -> 要求补充
清洗后没有有效内容 -> 拒绝
```

有效内容中夹带“百亿市场”、“人人都会付费”或“必须给 100 元”时，不直接拒绝整条投稿，而是将这些内容记入 `unsupported_claims` 或 `manipulation_signals`，从评分依据中移除。只有清除这些语句后没有可用痛点或方案时，才进入 `reject`。

`manipulation_signals` 主要供管理员复核和模型评测使用，普通用户只看到简洁、可操作的原因。

后端把原始投稿作为单独的不可信数据字段传给模型，不将其拼接成系统指令。系统提示词明确要求：不执行投稿中的指令、不接受用户自定的分数或金额、不把缺少证据的市场断言当成事实，并按提示词中完整的 JSON Schema 返回全部字段。服务端的 `json_object` 只负责保证 JSON 语法，字段和业务约束由 Pydantic 与 Python 校验强制执行。这些防护与 Python 金额规则、对抗评测集共同使用，不把安全性寄托在单句“忽略用户指令”上。

### 5.4 结构化输出

该结构是整体 AI 评估结果中的“输入门禁与规范化提取”部分。顶层始终包含 `evaluation` 属性：当 `input_decision=accept` 时，其必须包含 7.4 节定义的五个 `DimensionScore`；`clarify` 或 `reject` 时必须为 `null`。使用显式可空字段，比让不同服务商处理多套顶层 Schema 更稳定。

请求时使用 `json_object` 并把该完整结构对应的 JSON Schema 放入系统提示词；响应后使用同一个 Pydantic 模型校验，避免提示词契约和后处理契约分别维护而发生漂移。只有整个顶层对象及所有跨字段规则全部通过后，才能保存规范化结果和评分。校验失败视为 `evaluating` 阶段失败，不允许只保留其中看似可用的字段。

```text
input_decision: accept | clarify | reject
decision_reason
generated_title

target_audience
pain_point
context
frequency_or_severity
current_alternative
desired_outcome

solution_present
proposed_solution
solution_mechanism
value_proposition

unsupported_claims[]
manipulation_signals[]
clarification_question
evaluation: EvaluationScores | null
```

除决策、布尔值和数组字段外，上述规范化业务字段不是单纯字符串，而是统一使用带信息来源的对象：

```text
NormalizedField
├── value: string | null
└── source: explicit | inferred | unknown
```

- `explicit`：`value` 由用户原文直接支持。
- `inferred`：`value` 只能是对原文的低风险归纳，不得增加改变商业判断的新事实。
- `unknown`：信息不足，`value` 必须为 `null`，不得自动编造。

#### 门禁与用户反馈字段

| 字段 | 类型 | 解释与约束 |
| --- | --- | --- |
| `input_decision` | enum | 输入门禁结果。`accept` 继续估值和查重；`clarify` 要求用户补充；`reject` 终止本次评估。 |
| `decision_reason` | string | 简短、中性、可向用户展示的决策理由。不显示完整安全规则、模型提示词或内部风险分析。 |
| `generated_title` | string \| null | LLM 根据有效痛点生成的简短标题。`accept` 时必须提供，`clarify` 和 `reject` 时可为 `null`。标题不得包含“百亿市场”等未证实宣传。 |
| `clarification_question` | string \| null | 只有 `clarify` 时必填，提出一个最能解除当前不确定性的聚焦问题。`accept` 或 `reject` 时必须为 `null`。 |
| `evaluation` | `EvaluationScores \| null` | `accept` 时包含五个评分维度；`clarify/reject` 时必须为 `null`。详细子结构和权重见 7.4 节。 |

#### 共同痛点字段

| 字段 | 类型 | 解释与约束 |
| --- | --- | --- |
| `target_audience` | `NormalizedField` | 真正遇到该问题的人群或组织，不等同于方案的购买者。如果用户只写“我”，只能根据明确上下文谨慎归纳。 |
| `pain_point` | `NormalizedField` | 用一句中性话描述用户要解决的核心麻烦，不混入方案、市场规模或红包诉求。`accept` 时不得为 `unknown`。 |
| `context` | `NormalizedField` | 痛点出现的具体时间、地点、流程或使用场景。如果原文没有场景，应标记为 `unknown`。 |
| `frequency_or_severity` | `NormalizedField` | 用户明确描述的发生频率、时间/金钱损失或影响程度。“非常严重”等空泛形容词不能被改写成虚构数据。 |
| `current_alternative` | `NormalizedField` | 用户目前为解决或规避痛点而采取的做法，也可以是“目前没有办法”。原文未提及时不推测竞品或替代方案。 |
| `desired_outcome` | `NormalizedField` | 用户希望问题被解决后达到的结果。可以从明确的痛点做低风险反向归纳，但不得增加原文不支持的功能。 |

#### 解决方案字段

| 字段 | 类型 | 解释与约束 |
| --- | --- | --- |
| `solution_present` | boolean | 表示用户是否实际提出了可识别的解决思路。不能因为 LLM 自己想到了方案就设为 `true`。 |
| `proposed_solution` | `NormalizedField` | 对用户所提方案的中性摘要。`solution_present=false` 时必须为 `unknown/null`；为 `true` 时来源必须为 `explicit`，不得补写用户未提出的功能。 |
| `solution_mechanism` | `NormalizedField` | 方案通过什么核心机制解决痛点，例如自动化、信息匹配、预测或流程简化。只能归纳用户方案，不负责代用户设计新机制。 |
| `value_proposition` | `NormalizedField` | 用户方案相对 `current_alternative` 主张的改进，例如更快、更便宜或更容易使用。只提取价值主张，不把它自动视为已验证事实。 |

#### 风险与审计字段

| 字段 | 类型 | 解释与约束 |
| --- | --- | --- |
| `unsupported_claims` | string[] | 原文中无法从提交内容验证的市场规模、用户数量、付费意愿、成功率等断言。这些内容保留供审计，但不作为评分证据。 |
| `manipulation_signals` | enum[] | 检测到的诱导类型，可选值包括 `prompt_injection`、`score_or_amount_instruction`、`role_or_system_impersonation`、`irrelevant_padding` 和 `spam_or_gibberish`。数组可为空，不向普通用户展示完整内容。 |

#### 跨字段校验

Pydantic 在解析 LLM 输出后执行以下一致性校验：

- `accept` 时，`generated_title` 和 `pain_point.value` 必须存在，`clarification_question` 必须为 `null`。
- `accept` 时 `evaluation` 必须存在；`clarify/reject` 时必须为 `null`。
- `clarify` 时，`clarification_question` 必须存在且只询问一个聚焦问题。
- `reject` 时，不启动估值、Embedding 和语义查重。
- `solution_present=false` 时，三个方案字段必须全部为 `unknown/null`。
- `solution_present=true` 时，`proposed_solution.source` 必须为 `explicit`，期望结果不能被当成方案。
- 任何 `source=unknown` 的字段都必须同时满足 `value=null`。
- 每个评分维度的 `evidence_fields` 必须从九个规范化字段枚举中选择 1–3 个值。
- 输出中出现未声明字段时校验失败，不将其静默写入数据库。

LLM 只负责门禁判断、中性化提取和商业维度评分。最终金额始终由 Python 业务规则计算，模型无法直接越过金额上下限或重复折扣。

## 6. 语义查重

### 6.1 整体流程

查重不使用单一余弦相似度阈值直接决定结果，而是使用“Embedding 召回候选 + LLM 最终判定”的两阶段方案。

```text
用户原始描述
  -> 计算规范化文本哈希
  -> LLM 输入门禁、中性化提取和价值评分
  -> 用共同痛点字段生成 Embedding
  -> 完全相同：Python 判定高置信度 duplicate
  -> 否则：pgvector 精确检索 Top 10 历史候选
       -> 无候选：Python 判定 novel
       -> 有候选：LLM 一次性对比，判定 duplicate / related / novel
  -> Python 应用重复规则并计算红包
```

第一次 LLM 调用与商业价值评估共用，第二次 LLM 调用只负责短候选列表的查重判定。这样可以在查重前先移除用户的自夸、提示词注入和无关描述。

规范化文本哈希命中不跳过输入门禁和价值评分。新投稿通过门禁后，如果其哈希命中已完成、已通过门禁且未被管理员驳回的历史点子，后端还要再次比较两边的规范化文本以排除哈希冲突，然后可直接判定为高置信度 `duplicate`。该路径仍生成当前投稿的规范化结果、评分和 Embedding，但不再调用第二次 LLM。

如果 Embedding 或 LLM 查重失败，不能默认将投稿当作全新点子。该记录设为 `processing_status=failed`，记录 `failure_stage` 和安全错误码，暂不给出最终金额。

### 6.2 Embedding 内容

主查重向量仅使用各类投稿都具备的共同部分：

```text
目标用户 + 核心痛点 + 发生场景 + 期望结果
```

向量文本只使用 `explicit` 和低风险 `inferred` 字段，忽略 `unknown`。不将市场宣传、自我评价、对模型的指令或任何编造信息放入向量文本。方案内容不作为主向量的决定性部分，以避免相同痛点因解决方式不同而无法互相召回。方案是否提供新价值由后续 LLM 判定。

Embedding 服务与生成模型分开配置，均通过 OpenAI 兼容格式调用：

- `EMBEDDING_BASE_URL`
- `EMBEDDING_API_KEY`
- `EMBEDDING_MODEL_ID`
- `EMBEDDING_DIMENSIONS`
- `EMBEDDING_TIMEOUT_SECONDS`
- `EMBEDDING_MAX_RETRIES`

数据库必须记录 Embedding 模型、向量维度和输入格式版本。不同模型或输入版本产生的向量不混合比较；如果切换模型、维度或输入字段格式，需要重新生成全部有效历史点子的向量。

#### Embedding 服务能力探测

生产接入前使用独立脚本探测 OpenAI 兼容 `/embeddings` 端点。脚本将十条固定的中性中文文本作为一次批量请求发送，覆盖发票核对、食材保质期和预约爽约三个主题，以及相应的语义改写、同领域困难负样例和完全相同文本。探测输入只包含目标用户、痛点、场景和期望结果，不使用原始投稿或解决方案。

脚本严格校验响应数量、索引完整性、有限数值、非零向量、统一维度和可选的预期维度。完全相同文本的余弦相似度不得低于 `0.999`，该容差只用于发现服务输出严重不一致，不是语义查重阈值；精确重复仍由规范化文本哈希判定。每个主题的语义改写必须在排除自身和完全重复项后排名第一。该阶段只使用相对排名，不设置未经完整评测集校准的绝对语义阈值。

配置、HTTP 或向量结构失败使用退出码 `1`；结构合法但中文语义排序失败使用退出码 `2`。脚本默认只请求一次，可显式运行一到三次观察稳定性，不自动重试，也不打印完整向量或密钥。探测结果只用于筛选候选模型；模型、原生维度、成本和延迟确认后，才创建 `VECTOR(N)` 迁移和生产客户端。

2026 年 7 月通过 SiliconFlow 的 OpenAI 兼容接口实际探测 `BAAI/bge-m3`：批量输入和 float 向量格式可用，原生维度为 1024，返回向量范数为 1。三次探测中，每个主题的语义改写均稳定排名第一，正样例相对最佳负样例的差值约为 `0.0584–0.1381`，单次耗时约为 `0.40–0.79s`。完全相同文本的相似度为 `0.999888–0.999916`，说明服务输出存在极小浮点或推理波动，也证明原先 `0.9999` 的一致性门槛会产生假阴性，因此调整为 `0.999`。该小样例确认模型可以进入下一阶段，但不能替代 6.5 节的完整查重评测集。

生产接入固定使用 `VECTOR(1024)` 和 `embedding-input-v1`。AI 门禁接受后，后端只用目标用户、核心痛点、发生场景和期望结果构建中性文本，在数据库事务外调用 Embedding 服务；成功保存向量、配置模型、维度和输入版本并推进到 `checking_duplicate`，失败保留上游评估结果并记录 `failure_stage=embedding`。当前流水线会继续执行候选召回和最终查重判定，成功后进入 `completed`。

### 6.3 pgvector 检索

MVP 阶段使用 PostgreSQL + pgvector 的精确余弦距离查询，检索 Top 10 历史候选，不在小数据集上过早引入 HNSW 的近似召回损失。只在实际数据量和性能测试证明精确检索成为瓶颈时，再增加 HNSW 索引。

候选召回首先查询相同 `content_hash`、`internal_id` 更小且已经完成处理的历史点子，并重新比较经过 NFKC、大小写归一化和空白压缩后的完整原文，防止理论上的哈希碰撞。命中时返回最早的内部 ID，并跳过向量检索；精确路径不要求 Embedding 版本一致。

精确路径未命中时才运行向量查询。语义候选必须与当前记录使用相同的 Embedding 模型、1024 维度和输入版本，按余弦距离升序、内部 ID 升序稳定排序。服务返回 `1 - cosine_distance` 作为内部相似度和经正式 Pydantic 契约重新校验的规范化内容；相似度只用于排序和后续评测，不直接决定重复结论。损坏的历史 JSONB 不会被静默跳过。

当前召回只包含 `internal_id` 更小、已完成、`input_decision=accept` 且具有完整查重快照的历史点子，不允许待查重记录互相引用。候选会排除 `effective_duplicate_verdict=duplicate` 的记录，避免形成重复链；`novel` 和存在新增价值的 `related` 仍可作为历史候选。管理员驳回过滤留到审核字段落库后补充。

召回仍是内部服务，不通过 HTTP 暴露候选、相似度或内部 ID。投稿流水线优先处理精确哈希；没有语义候选时直接保存高置信度 `novel`，有候选时才调用生成模型，并将最终快照整体写入数据库。

余弦相似度在本方案中只负责排序和召回候选，不使用拍脑袋的全局阈值直接决定是否重复。

### 6.4 痛点与方案的最终判定

LLM 需要分开判定痛点关系和方案关系：

```text
pain_relation: same | related | different
solution_relation: same | related | different | not_applicable
verdict: duplicate | related | novel
matched_internal_id
same_aspects[]
different_aspects[]
added_value
confidence: high | medium | low
reason
```

具体规则：

| 痛点关系 | 方案关系 | 最终判定 |
| --- | --- | --- |
| 相同 | 两边都没有方案 | `duplicate` |
| 相同 | 方案实质相同 | `duplicate` |
| 相同 | 新投稿有明显不同的方案 | `related` |
| 相同 | 历史只有痛点，新投稿增加具体方案 | `related` |
| 相关但不相同 | 任意 | `related` |
| 不同 | 任意 | `novel` |

目标用户名称不同但未改变实际场景、约束和付费方时，不应被当成新痛点。如果用户群差异实质改变了场景、解决方式或经济买单方，则可以判为 `related` 或 `novel`。

LLM 返回的 `matched_internal_id` 必须来自本次 Top 10 候选列表，后端需要再次验证，不接受模型生成的任意 ID。对普通用户返回结果时，再将它转换为对应的 `matched_public_id`。

`duplicate` 或 `related` 必须返回 `matched_internal_id`，`novel` 时必须为 `null`。候选列表为空时后端直接以 `duplicate_method=no_candidates` 判为高置信度 `novel`，不调用第二次 LLM。

查重契约版本固定为 `duplicate-evaluation-v1`。发送给模型的内容只包含规范化后的目标用户、痛点、场景、频率、现有替代、期望结果和方案字段，不包含用户原文、商业评分、余弦相似度、Embedding 或诱导审计字段。候选使用内部 ID，模型输出后由 Pydantic 结合本次候选列表再次校验，不接受任意生成的 ID。`same_aspects` 和 `different_aspects` 使用固定公开字段枚举且不能重叠。

独立探测脚本使用三组 `duplicate`、三组 `related` 和三组 `novel` 合成中文案例检查结构、关系和最佳候选。脚本复用现有 `AI_*` 配置，逐案例调用且不自动重试；生产流水线则按 `AI_MAX_RETRIES` 自动处理暂时性错误和非法结构，并保存通过同一正式契约校验的查重快照。

2026 年 7 月使用 `mimo-v2.5-pro` 进行真实探测。初始案例暴露了“一方没有方案时应使用 `different`”以及相关痛点边界描述不够明确的问题；补充关系定义并消除案例歧义后，九个固定案例全部通过，单轮总耗时 `36.10s`，总 Token 用量 `14506`，服务端未返回请求 ID。该结果只表示模型能够处理当前明确的小样本，不代表生产数据上的准确率或稳定性。

只有 `confidence=high` 的 `duplicate` 才应用重复红包规则。中低置信度的 `duplicate` 在金额计算时保守地按 `related` 处理，同时保留 AI 原始判定供管理员复核。`related` 向用户提示“发现相关点子，但存在新增价值”，不直接按重复扣减金额。

### 6.5 查重评测集

面试 MVP 先使用九个代表性中文案例验证基本契约和判定矩阵，包括：

- 明确的改写重复。
- 同领域但不同痛点的困难负样本。
- 相同痛点但不同解决方案。
- 目标用户变化造成的实质差异。
- 否定词、时间、地点、规模或付费方变化。
- 加入自夸、无关文字和提示词注入后的同一投稿。

九个案例只用于发现明显的结构和行为问题，不把小样本通过率包装成统计指标。如果项目实际准备上线，再扩展为 40–60 对以上的人工标注数据，并评估候选召回 `Recall@10`、重复判定 Precision/Recall、稳定性、调用成本和延迟。由于误判重复会直接伤害用户信任，最终自动扣减应优先保证 Precision。

### 6.6 匹配点子查询与用户复核

查重结果不能只告诉用户“之前有人提交过”。当结果为 `duplicate` 或 `related` 时，后端需要返回匹配点子的随机公开 ID 和可查询的脱敏摘要，让用户能够核对判定。

#### 内部 ID 与公开 ID

点子同时保留：

- `internal_id`：数据库内部主键，不通过任何 HTTP API 暴露。
- `public_id`：随机 UUIDv4，用于向其他已登录用户引用某条点子。

不使用可遍历的连续数字公开 ID。数据库用 `matched_idea_id` 同时表达 `related` 和 `duplicate` 的最佳匹配对象，不使用只适用于“重复”的 `duplicate_of_id`。当前候选查询直接排除有效判定为 `duplicate` 的历史记录，因此新结果只会指向非重复历史点子，不形成重复链条。

#### 数据视图与接口

```text
GET /api/me/ideas/{public_id}
  -> 当前用户查看自己的完整投稿和评估结果

GET /api/ideas/{public_id}/summary
  -> 已登录用户跨用户查看脱敏公开摘要

GET /api/admin/ideas/{public_id}
  -> 管理员查看原始投稿、AI 结果、查重信息和审核信息
```

跨用户摘要接口必须使用独立的 Pydantic 响应模型 `PublicIdeaSummary`，不得直接序列化 ORM 实体。

```text
PublicIdeaSummary
├── public_id
├── generated_title
├── target_audience
├── pain_point
├── context
├── solution_present
├── solution_outline
└── created_date
```

`solution_outline` 只显示解决机制的中性概括，不返回原始投稿。公开摘要不接受 LLM 任意生成的字段，而是由后端从已校验的规范化结果中按 `PublicIdeaSummary` 白名单投影，再移除手机号、邮箱、身份证号、详细地址等个人信息。某个字段仍疑似含有敏感信息时，宁可将该字段置空，不回退到原文。

公开摘要明确不返回：

- 投稿人的用户 ID、用户名和联系方式。
- 原始投稿文本。
- 红包金额、账户累计金额、模拟打款状态和流水号。
- AI 内部评分、提示词和置信度。
- `unsupported_claims`、`manipulation_signals`、Embedding 和余弦相似度。
- 管理员备注和审核记录。

#### 查重结果响应

向用户展示的查重结果包含：

```text
verdict
matched_public_id
matched_idea_url
same_aspects[]
different_aspects[]
reason
commercial_score
base_amount
duplicate_deduction
final_amount
```

`same_aspects` 和 `different_aspects` 只允许引用可公开比较的规范化字段，例如 `target_audience`、`pain_point`、`context` 和 `solution_mechanism`。用户可以通过 `matched_idea_url` 查看对方的公开摘要，不依赖投稿日期作为重复证据。

`duplicate_deduction` 表示自动规则计算得到的重复扣减。管理员调整只更新 `admin_amount` 和用户当前看到的 `final_amount`，不会覆盖基础金额或自动扣减。

## 7. 红包评分与金额规则

### 7.1 评分原则

输入质量、商业价值和平台查重是三套独立判定：

- 输入质量只决定 `accept | clarify | reject`，不因为字数多而加分。
- 商业价值产生 `commercial_score` 和 `base_amount`。
- 查重结果在基础金额上应用重复调整。

LLM 只对每个维度输出 0–5 的整数分、简要理由和证据字段。总分、基础金额、重复调整和最终金额全部由 Python 使用 `Decimal` 计算。

### 7.2 评分维度和权重

| 维度 | 权重 | 核心问题 |
| --- | ---: | --- |
| `demand_breadth` 需求广度 | 20 | 有多少人或组织真实面临该问题？ |
| `pain_intensity` 痛点强度 | 25 | 发生多频繁，会造成多大的时间、金钱、风险或精力损失？ |
| `willingness_to_pay` 付费意愿 | 25 | 是否有明确买单方，解决后是否能带来可解释的收益或避损？ |
| `feasibility` 可解决性/方案可行性 | 15 | 这个问题是否可被实际干预，用户提出的方案是否现实？ |
| `novelty` 新颖性/差异化 | 15 | 痛点洞察或解决机制相比常见做法是否提供了新增价值？ |

痛点投稿和带方案投稿使用同一套权重。没有方案时，`feasibility` 评估痛点的可行动性，`novelty` 评估痛点洞察和场景的新增价值；提供了方案也不自动加分，不现实的方案可以拉低可行性。

### 7.3 维度分数锚点

2 分和 4 分作为相邻锚点之间的中间档，关键锚点如下：

| 维度 | 0 分 | 1 分 | 3 分 | 5 分 |
| --- | --- | --- | --- | --- |
| 需求广度 | 无法识别目标用户 | 极少数人且极偶发 | 明确细分市场中的常见问题 | 大众或大量组织长期普遍存在 |
| 痛点强度 | 无明确负面影响 | 轻微不便，容易忽略 | 经常消耗时间、金钱或精力 | 高频且紧迫，可能导致重大损失或风险 |
| 付费意愿 | 无可识别买单方 | 仅属于“有了更好” | 可明确节省时间/成本，买单方基本清楚 | 直接影响收入、重大成本、风险或合规责任 |
| 可行性 | 目标矛盾、明显违法或无法实现 | 依赖当前不存在的关键能力 | 使用现有技术和资源基本可行 | 路径简单直接、依赖少、容易验证 |
| 新颖性 | 没有任何新增认识 | 常见问题或标准方案 | 针对明确场景/人群/流程有新角度 | 罕见且可信的新洞察或明显不同的解决机制 |

`unsupported_claims` 和 `manipulation_signals` 不得作为加分证据。如果核心信息不足到无法使用锚点评分，应进入 `clarify`，不是给出一个随意的低分。

### 7.4 评分结构和总分

```text
DimensionScore
├── score: 0 | 1 | 2 | 3 | 4 | 5
├── reason: string
├── confidence: high | medium | low
└── evidence_fields: string[]

EvaluationScores
├── demand_breadth: DimensionScore
├── pain_intensity: DimensionScore
├── willingness_to_pay: DimensionScore
├── feasibility: DimensionScore
└── novelty: DimensionScore
```

`confidence` 用于管理员复核和模型评测，不直接乘到金额上。`evidence_fields` 只能引用已声明的规范化字段，后端要验证引用是否合法。

```text
commercial_score =
  demand_breadth.score / 5 * 20
+ pain_intensity.score / 5 * 25
+ willingness_to_pay.score / 5 * 25
+ feasibility.score / 5 * 15
+ novelty.score / 5 * 15
```

由于权重均可被 5 整除，`commercial_score` 最终是 0–100 的整数。

### 7.5 基础红包曲线

```text
commercial_score <= 30:
    base_amount = 0

commercial_score > 30:
    base_amount = 100 * ((commercial_score - 30) / 70) ^ 2
```

金额使用 `Decimal` 以 `ROUND_HALF_UP` 保留两位小数，并强制限制在 ¥0–¥100。

| 商业分 | 基础红包 |
| ---: | ---: |
| 30 | ¥0.00 |
| 40 | ¥2.04 |
| 50 | ¥8.16 |
| 60 | ¥18.37 |
| 70 | ¥32.65 |
| 80 | ¥51.02 |
| 90 | ¥73.47 |
| 100 | ¥100.00 |

### 7.6 重复调整和最终金额

| 状态 | AI 金额规则 |
| --- | --- |
| `novel` | `final_amount = base_amount` |
| `related` | `final_amount = base_amount` |
| `duplicate` 且 `confidence=high` | `final_amount = 0` |
| `duplicate` 且中/低置信度 | 计算时按 `related` 处理，保留 AI 原判定供审核 |
| `clarify` / `reject` | 不生成基础或最终红包 |
| AI / Embedding / 查重失败 | 暂不生成最终红包，等待重试 |
| 管理员驳回 | `final_amount = 0` |

```text
duplicate_deduction = base_amount - final_amount
```

重复投稿不保留小额鼓励金，避免用户通过批量改写历史点子获取金额。页面同时显示基础价值、重复扣减、最终金额和匹配点子的脱敏查询入口；管理员调整后 `final_amount` 表示当前有效金额。

## 8. 点子状态机与核心数据模型

### 8.1 相互独立的状态维度

处理进度、输入结论、查重结论和管理员最终动作分别保存；模拟打款状态由管理员动作和流水是否存在推导。

| 字段 | 可选值 | 责任 |
| --- | --- | --- |
| `processing_status` | `pending | evaluating | embedding | checking_duplicate | completed | failed` | 表示外部 AI/Embedding 处理进度 |
| `input_decision` | `null | accept | clarify | reject` | 表示输入是否足以进入估值和查重 |
| `ai_duplicate_verdict` | `null | novel | related | duplicate` | 保留 LLM 的原始查重判定 |
| `effective_duplicate_verdict` | `null | novel | related | duplicate` | 实际用于金额计算的判定，可被保守规则或管理员覆盖 |
| `admin_action` | `null | confirmed | adjusted | rejected` | 管理员一次性最终处理结论；非空后锁定 |

AI 返回中/低置信度 `duplicate` 时，`ai_duplicate_verdict=duplicate`，但 `effective_duplicate_verdict=related`，从而保留原始信息又避免自动误扣。

### 8.2 处理状态转换

```text
pending
  -> evaluating
       -> completed             (input_decision=clarify/reject)
       -> embedding             (input_decision=accept)
            -> checking_duplicate
                 -> completed

pending/evaluating/embedding/checking_duplicate
  -> failed                     (记录 failure_stage/failure_code)

failed
  -> evaluating                 (failure_stage=evaluating)
  -> embedding                  (failure_stage=embedding)
  -> checking_duplicate         (failure_stage=checking_duplicate)
```

MVP 采用同步提交流程，不引入 Celery/Redis 任务队列。后端在调用外部服务前先创建 `ideas` 记录，并在每个阶段更新 `processing_status`。前端显示处理中状态；超时或失败时保留同一个点子 ID 和安全错误码，重试不创建新投稿。重试从 `failure_stage` 继续，复用已通过校验的上游结果，避免重复花费和模型随机性改变。

每次外部调用前先用短事务提交新状态，不在等待 AI 时长时占用数据库事务。重试使用条件更新，只允许一个请求把当前 `failed` 状态转回对应阶段，其他并发重试直接返回已有状态。每次成功发起重试时 `retry_count + 1`。

同步流水线只继续处理当前请求亲自推进的阶段。幂等或并发重放看到已有 `evaluating`、`embedding` 或 `checking_duplicate` 时只返回快照，不重复调用外部服务。MVP 暂不增加任务租约；进程在阶段间被强制终止造成的遗留处理中记录，后续由统一恢复任务处理。

```text
POST /api/me/ideas/{public_id}/retry
```

该接口只允许点子所有者对 `processing_status=failed` 的记录调用，不接受前端指定重试阶段。后端始终使用已保存的 `failure_stage`。

客户端每次新建提交生成 `submission_key` UUID，后端对 `(user_id, submission_key)` 建立唯一约束，防止浏览器重试或用户双击导致重复执行外部 AI 调用。

投稿入口首先通过 `POST /api/me/ideas` 持久化 `pending` 记录。首次创建返回 `201`；
相同用户使用相同 `submission_key` 重放完全一致的原文时返回原记录和 `200`，不重复创建；
如果原文有任何变化则返回 `409`。幂等判断比较完整原文，规范化 `content_hash` 只用于后续查重召回。
个人投稿通过 `GET /api/me/ideas?limit=20&offset=0` 查询，采用 `offset + total` 分页，
按 `created_at DESC, internal_id DESC` 稳定排序；个人详情接口始终同时限定当前用户和 `public_id`。

`clarify` 和 `reject` 是该次投稿的终态。`clarify` 时前端保留原文并显示一个补充问题，用户修改后作为新投稿提交；MVP 不引入多轮对话和消息表。

管理员只处理 `completed + accept` 投稿，执行一次 `confirmed | adjusted | rejected`。确认或调整后的正金额在同一事务中生成模拟流水；0 元和驳回不生成流水。管理员动作一旦写入即为终态，不实现用户复核申请、多次审核或审核历史表。

用户响应中的 `payout_status` 不单独落库：处理中为 `not_ready`，完成但尚未处理为 `awaiting_admin`，存在模拟流水为 `confirmed`，管理员处理后没有流水为 `not_applicable`。

### 8.3 `ideas` 当前快照表

`ideas` 表保存当前可查询状态；大块、可演进的 LLM 结构用 JSONB 保存，需要列表筛选、约束和聚合的字段则单独建列。

| 分组 | 字段 | 说明 |
| --- | --- | --- |
| 标识与归属 | `internal_id`, `public_id`, `user_id`, `submission_key` | `internal_id` 供内部关联，`public_id` 为 UUIDv4，`submission_key` 与用户联合唯一 |
| 原始输入 | `raw_content`, `content_hash` | 保存原文；`content_hash` 基于 Unicode 归一化、空白压缩和大小写归一化后的文本计算 |
| 处理进度 | `processing_status`, `input_decision`, `failure_stage`, `failure_code`, `retry_count` | `failure_stage` 为 `evaluating \| embedding \| checking_duplicate`；失败只保存安全错误码，完整服务商错误留在脱敏日志中 |
| 规范化结果 | `normalized_content` | 使用 JSONB 保存通过 Pydantic 校验的规范化对象；公开摘要在读取时按独立白名单模型投影并脱敏，不重复保存 |
| 估值结果 | `dimension_scores`, `commercial_score`, `base_amount`, `evaluation_model`, `evaluation_prompt_version`, `evaluation_schema_version` | 五个维度的原始输出保存为 JSONB，总分和金额单独建列；模型、提示词和 Schema 版本用于复现结果 |
| 向量 | `embedding`, `embedding_model`, `embedding_dimensions`, `embedding_input_version` | 四个字段必须同时存在，一个部署只使用一组兼容的模型、维度和输入格式 |
| 查重快照 | `duplicate_method`, `ai_duplicate_verdict`, `effective_duplicate_verdict`, `duplicate_confidence`, `matched_idea_id`, `duplicate_reason`, `duplicate_comparison`, `duplicate_model`, `duplicate_prompt_version`, `duplicate_schema_version` | `duplicate_method` 为 `exact_hash \| no_candidates \| llm_candidates`；`matched_idea_id` 只指向更早的非重复点子 |
| 金额快照 | `commercial_score`, `base_amount`, `duplicate_deduction`, `final_amount` | 当前 MVP 保存规则计算结果；所有金额使用 `NUMERIC(6,2)`，不使用浮点数 |
| 管理员处理 | `admin_action`, `admin_amount`, `admin_reason`, `processed_by_admin_id`, `admin_processed_at` | 保存一次性最终动作；调整和驳回必须有理由 |
| 模拟打款 | 见 `simulated_payouts` | 只为管理员确认后的正金额生成一条虚拟流水 |
| 时间 | `created_at`, `updated_at`, `completed_at` | 统一使用带时区时间，存储 UTC，展示时按用户时区转换 |

`internal_id` 使用 `BIGINT` 自增主键，`public_id` 和 `submission_key` 使用 PostgreSQL `UUID`，`raw_content` 使用 `TEXT`，`content_hash` 使用 64 位 SHA-256 十六进制字符串。状态列使用 `VARCHAR + CHECK` 约束和 Python Enum，避免在 MVP 中为修改 PostgreSQL Enum 增加迁移复杂度。时间列使用 `TIMESTAMPTZ`。

`failure_code` 只使用可控枚举，例如 `provider_config_error`、`provider_auth_error`、`json_mode_unsupported`、`provider_timeout`、`provider_rate_limited`、`invalid_ai_response`、`invalid_ai_output`、`embedding_dimension_mismatch` 和 `provider_error`，不直接写入服务商响应文本。`retry_count` 默认为 0，普通用户对同一点子最多成功发起 3 次阶段重试，避免失效配置无限消耗外部 API 费用。

`commercial_score` 使用 0–100 整数，金额列使用 `NUMERIC(6,2)`。`normalized_content`、`dimension_scores` 和 `duplicate_comparison` 写入前必须先通过对应 Pydantic 模型，不将未校验的模型输出直接写入 JSONB。`PublicIdeaSummary` 在读取时从已校验规范化结果投影，并逐字段移除手机号、邮箱、身份证号和明显详细地址。

`embedding` 当前固定为 `VECTOR(1024)`，与已探测的 `BAAI/bge-m3` 一致；`EMBEDDING_DIMENSIONS` 必须为 1024。`embedding_input_version` 当前为 `embedding-input-v1`。切换到不同维度的模型需要 Alembic 迁移，切换模型或输入版本也必须为全部有效点子重建向量；重建完成前不混合查询新旧向量。

必要索引和约束为：`public_id` 唯一、`(user_id, submission_key)` 唯一、`(user_id, created_at)` 用于个人历史、`(processing_status, created_at)` 用于处理查询、`matched_idea_id` 用于查找关联记录。`content_hash` 使用普通索引加速完全相同内容检查，但不设唯一约束。MVP 数据量下后台直接倒序分页，不为管理员动作增加额外索引，精确向量检索也不创建 HNSW 索引。

### 8.4 管理员最终处理

管理员通过以下接口查看全部投稿、单条详情和基础汇总：

```text
GET /api/admin/ideas
GET /api/admin/ideas/{public_id}
GET /api/admin/summary
```

最终处理接口为：

```text
POST /api/admin/ideas/{public_id}/process
action: confirmed | adjusted | rejected
amount: 仅 adjusted 使用
reason: adjusted/rejected 必填
```

`confirmed` 使用规则计算金额，`adjusted` 使用管理员提供的 ¥0–¥100 金额，`rejected` 将最终金额置为 0。只允许管理员处理 `completed + accept` 投稿，处理后不能再次修改。当前 MVP 不实现用户复核申请、多次审核、判定覆盖或审核历史表。

### 8.5 模拟打款记录

`simulated_payouts` 保存管理员最终处理时生成的一对一虚拟流水：

```text
simulated_payouts
├── id
├── idea_id -> ideas.internal_id (unique)
├── amount: NUMERIC(6,2)
├── reference: string (unique)
├── confirmed_by_admin_id -> users.id
└── confirmed_at: TIMESTAMPTZ
```

`reference` 使用以 `SIM-` 开头的随机流水号，只用于演示，不映射任何真实支付渠道。管理员确认或调整后的金额大于 0 时，服务在同一事务内锁定点子、写入最终动作并创建流水；金额为 0 或驳回时不创建。`idea_id` 唯一约束保证每条点子最多一条模拟流水。

用户详情只返回金额、流水号和确认时间，不返回处理管理员 ID。流水金额来自服务端最终金额，不接受客户端指定流水号、确认人或确认时间。

### 8.6 数据一致性约束

- 当前阶段 `processing_status=completed` 且 `input_decision=accept` 时，规范化结果、评分、Embedding、查重判定和红包估值必须存在。
- `input_decision=clarify/reject` 时，不允许存在评分、Embedding、查重结果和金额。
- `duplicate_method=exact_hash` 时必须为高置信度 `duplicate` 且存在 `matched_idea_id`；`duplicate_method=no_candidates` 时必须为高置信度 `novel`。这两种路径的 `duplicate_model` 为 `null`。
- `duplicate_method=llm_candidates` 时，`duplicate_comparison`、`duplicate_model`、`duplicate_prompt_version` 和 `duplicate_schema_version` 必须存在。
- `effective_duplicate_verdict=duplicate` 时，`final_amount=0` 且 `duplicate_deduction=base_amount`。
- `effective_duplicate_verdict=novel/related` 时，`final_amount=base_amount` 且 `duplicate_deduction=0`。
- `effective_duplicate_verdict=related` 时，`matched_idea_id` 必须存在；`effective_duplicate_verdict=novel` 时，`matched_idea_id` 必须为 `null`。
- `processing_status=failed` 时，`failure_stage` 和 `failure_code` 必须存在，`final_amount` 不得生成。
- 管理员动作、金额、处理人和处理时间必须同时为空或同时存在；调整和驳回必须保存理由。
- `admin_action=rejected` 时 `final_amount=0`；其他调整必须在 ¥0–¥100 范围内。
- `final_amount` 始终由服务端规则或管理员审核流程写入，不接受前端直接传值。
- 只有管理员最终处理后的正金额才创建 `simulated_payouts`；`idea_id` 唯一且流水金额等于锁定后的 `final_amount`。
- 不接受前端传入模拟流水号、确认人或确认时间。

两个高度相似投稿在极短时间内并发提交时，仍可能在对方完成查重前同时被判为全新。由于 MVP 不真实发放资金，本期将它记为已知竞态条件，交由管理员复核；真实付款前需要增加二次最终查重或串行化结算。

## 9. 数据校验边界

Pydantic 用于保护六类数据边界：

1. **账户与认证**：用户名和密码的格式、长度及必填约束。
2. **用户提交**：必填字段、字符串长度和数据类型。
3. **AI 结果**：评分范围、必填理由、重复判定和关联记录。
4. **管理员操作**：调整后的金额范围、审核状态和驳回理由。
5. **跨用户公开摘要**：只返回允许列表中的脱敏字段，不暴露原始投稿和用户信息。
6. **模拟打款确认**：请求只允许可选的管理员内部备注，金额、流水号、确认人和时间全部由后端生成。

该校验层不替代业务规则。例如“重复点子应如何折扣”属于业务规则，而“金额不能为负数”属于数据校验。

## 10. 数据库迁移

Alembic 用于管理 PostgreSQL 表结构的版本变化，不是用于在两台服务器之间搬运数据。

示例：开发后台复核时，需要为已有的点子表新增审核状态和驳回理由。Alembic 迁移可以在保留已有数据的同时修改表结构。

本 MVP 至少保留一个初始迁移，后续每次表结构变更都通过新迁移记录，不在部署环境中手工改表。

## 11. 用户认证、授权与会话

本 MVP 实现本地用户名/密码认证，不接入 OAuth，也不引入 Redis。普通用户和管理员使用同一套账户与会话机制，以 `role` 字段区分权限。

### 11.1 核心数据关系

```text
users
├── id
├── username (unique)
├── password_hash
├── role: user | admin
├── status
└── created_at

sessions
├── id
├── token_hash (unique)
├── user_id -> users.id
├── expires_at
├── revoked_at
└── created_at

ideas
├── internal_id
├── public_id: UUIDv4 (unique)
├── user_id -> users.id
└── matched_idea_id -> ideas.internal_id (nullable)

simulated_payouts
├── idea_id -> ideas.internal_id (unique)
└── confirmed_by_admin_id -> users.id
```

每个点子、AI 评估结果、最终红包金额和模拟打款都通过 `ideas.user_id` 归属到提交用户。后端根据当前会话确定用户，创建点子时不接受前端传入的 `user_id`；模拟打款的确认人同样只能来自当前管理员会话。

### 11.2 最小认证闭环

本期至少包含：

- 用户名和密码注册。
- 用户名和密码登录。
- 获取当前登录用户。
- 退出登录并撤销当前会话。
- 普通用户查看自己的提交和金额。
- 已登录用户通过公开 ID 查看他人点子的脱敏摘要。
- 普通用户对自己被判重的点子申请管理员复核。
- 管理员查看全部用户的提交并执行复核。
- 管理员对已完成前置流程的正金额点子确认模拟打款。

公开注册始终创建 `user` 角色，不接受前端传入的 `role`。第一个管理员账号通过受控的一次性命令创建，不提供公开的“注册管理员”接口。

### 11.3 密码与会话

密码只在 PostgreSQL 中保存 Argon2id 哈希，不保存或记录明文密码。

登录成功后，后端生成高强度随机 Session Token：

1. 原始 Token 只发送给浏览器，存入 Cookie。
2. PostgreSQL 只保存 Token 的哈希、所属用户和过期/撤销时间。
3. 每次受保护请求都通过 Cookie 查找会话和当前用户。
4. 退出登录、会话过期或账号停用后，服务端拒绝该会话。

会话采用固定 3 天有效期，不做滑动续期。每个用户在 `sessions` 表中最多保留一行；
注册成功会直接建立会话，之后每次成功登录都会在数据库事务中锁定用户并覆盖原会话。
因此后一次登录成功后，旧设备持有的 Cookie 会立即失效，系统不允许同一账号多端同时登录。
退出登录只撤销当前会话，新登录可以复用并覆盖这条已撤销记录，不保留历史会话列表。

生产环境的 Cookie 使用 `Secure`、`HttpOnly`、`SameSite=Strict` 和 `Path=/`，且会话不写入 `localStorage` 或 `sessionStorage`。后台的状态变更请求限制同源访问，登录和注册接口进行基本的频率限制。

### 11.4 不使用 Redis 的原因

本项目是低流量、单个 API 实例的 MVP，PostgreSQL 已经是必需依赖，完全可以承担用户会话查询。把会话放入 PostgreSQL 还可以直接支持退出、账号停用和单会话撤销。

只在后续出现多实例高并发、PostgreSQL 会话查询成为实际瓶颈，或确实需要 Redis TTL 和集中会话能力时，再引入 Redis。

### 11.5 MVP 账户限制

本期不实现邮箱/手机号验证、找回密码、实名认证、多账号防刷和提现风控。因为不发放真实资金，这些可以作为明确的 MVP 限制；在接入真实红包前必须补齐。

## 12. 管理后台与汇总口径

管理员列表按最新提交倒序分页，显示点子标题、用户、处理状态、商业分、重复判定、最终金额和管理员动作，点进详情后查看原文、AI 理由、匹配对比并执行最终处理。MVP 不实现复杂筛选。

页面顶部提供与当前时间筛选一致的汇总卡片：

- 提交总数。
- 已完成且通过门禁的点子数。
- 当前有效重复数。
- 当前最终估值总额。
- 已确认模拟打款笔数和总额。

最终估值总额只求和非空 `ideas.final_amount`，使用管理员覆盖后的当前快照；已确认模拟打款总额只求和 `simulated_payouts.amount`。两者必须分开展示，且都不表示真实支付。MVP 直接对 `ideas` 和 `simulated_payouts` 执行 SQL 聚合，不新建汇总表、定时任务或 Redis 缓存。

## 13. 页面与交互闭环

普通用户页面保持小而完整：

| 路由 | 主要内容 |
| --- | --- |
| `/register` | 用户名和密码注册，成功后进入登录或直接建立会话 |
| `/login` | 用户和管理员共用的登录页 |
| `/` | 单文本框提交页，同时展示输入建议和当前用户最近提交 |
| `/ideas/{public_id}` | 当前用户的完整结果页，包含金额、评分理由、查重对比和模拟打款状态 |
| `/ideas` | 当前用户的提交历史，显示状态、标题、日期、最终金额和模拟打款状态 |
| `/matches/{public_id}` | 其他点子的脱敏公开摘要，只能从查重结果进入 |
| `/admin` | 汇总卡片、筛选器和点子列表 |
| `/admin/ideas/{public_id}` | 管理员详情、AI 结果、匹配对比，以及确认、调价或驳回操作 |

提交后前端立即进入该点子的结果页，并根据状态展示不同界面：

- `pending/evaluating/embedding/checking_duplicate`：显示简短阶段文案和进度状态，不伪造金额。
- `completed + accept`：最突出地显示 `final_amount`，同时显示基础金额、AI 重复扣减、是否经管理员调整、五个评分维度及理由。
- `completed + clarify`：显示唯一补充问题，将原文带回提交框供用户修改后新建提交。
- `completed + reject`：显示中性拒绝理由，不显示内部风险信号或安全提示词。
- `failed`：显示安全错误码和重试按钮；超过重试上限时引导用户稍后再试，不暴露服务商响应。

`completed + accept` 还要展示独立的模拟打款进度：

- `not_ready`：“投稿尚未完成处理”。
- `awaiting_admin`：“等待管理员最终处理”。
- `confirmed`：“模拟打款已确认”，同时显示确认金额、模拟流水号和确认时间。
- `not_applicable`：“无需模拟打款”。

`duplicate` 结果使用明确的“高度重复”文案和 0 元结果；`related` 显示“发现相关点子，但你的内容有新增价值”，不使用惩罚性红色警告。两者都展示可公开的相同点、不同点和匹配摘要链接。页面在确认前只表述“估值金额”；确认后只能使用“模拟打款已确认”，始终不使用“已发放”、“已到账”或“支付成功”等真实支付语义。

管理员详情页只在 `admin_action=null` 时显示“确认模拟打款”“调整后确认”和“驳回”。处理后将控件替换为只读最终结果和可选模拟流水。

除注册和登录外，普通用户路由都需要登录，管理路由额外要求 `role=admin`。后端权限校验是真实安全边界，前端隐藏菜单只用于交互。所有页面使用 Tailwind 实现基本响应式，至少在桌面和手机宽度下可用。

## 14. 测试策略

### 14.1 Python 单元测试

使用 pytest 重点验证：

- 各评分维度到基础金额的计算。
- 重复判定对红包的折扣规则。
- 金额上下限和边界值。
- AI 返回不完整或非法数据时的处理。
- JSON mode 返回合法 JSON 但字段缺失、额外字段、类型错误或跨字段矛盾时必须整体失败。
- 点子处理、输入、查重、审核和模拟打款状态的合法转换与不变式。
- 同一 `(user_id, submission_key)` 不会重复创建点子或重复调用 AI。
- 失败重试从 `failure_stage` 继续，并发重试只有一个能获得处理权。
- 完全相同内容会优先加入候选，但不跳过输入门禁。
- 管理员调整和驳回的状态流转。
- 用户注册、密码校验、会话过期和撤销。
- 普通用户不能访问他人提交或管理员接口。
- 跨用户摘要只返回脱敏允许字段，不暴露原始投稿、所有者或金额。
- 用户只能为自己的点子申请复核，且管理员修改判定后金额被正确重算。
- 只有已完成、已复核且正金额的点子能进入待打款，0 元、驳回和未处理记录不能确认。
- 并发或重复确认只生成一笔模拟打款，金额来自服务端快照，确认后无法再修改点子金额或判定。
- 普通用户不能调用模拟打款确认接口，管理员也不能传入自定金额、流水号或确认人。
- 后台汇总的重复率分母、空金额排除、管理员覆盖后金额口径，以及最终估值与已确认模拟打款的分离统计。

### 14.2 API 测试

使用 pytest + HTTPX 测试 FastAPI 接口。AI 服务在自动测试中使用 mock，避免消耗额度、网络波动和模型随机性导致测试不稳定。

### 14.3 端到端测试

使用 pytest-playwright 通过 Python 驱动浏览器，至少覆盖：

- 用户提交点子并查看评估结果。
- 用户注册、登录、查看自己的提交并退出。
- 用户提交语义重复点子并看到明确提示。
- 用户通过匹配点子 ID 查看脱敏摘要，并对误判提交复核申请。
- 管理员查看记录、调整金额和驳回记录。
- 管理员完成评估确认后执行模拟打款确认，用户随后看到模拟流水号和确认金额。

## 15. 部署方向

初始演示环境计划部署在国外 2 核 2 GB 服务器，避免未备案域名部署到中国内地服务器的问题。国内 4 核 4 GB 服务器暂不作为公开演示环境。

预计的容器结构：

```text
Nginx
├── /       -> React 静态文件
└── /api/*  -> FastAPI
                  └── PostgreSQL
```

部署约束：

- 使用 Docker Compose 管理服务。
- Nginx 监听 80/443，使用 Certbot 申请和续期 Let's Encrypt 证书；前端与 API 使用同一域名和同源路径。
- PostgreSQL 使用持久化数据卷。
- AI 和 Embedding 使用外部 API，不在 2 GB 服务器上运行本地模型。
- 生产服务器不运行 Playwright 浏览器测试。
- 前端尽量不依赖国内访问不稳定的字体或 CDN。
- 密钥和数据库密码通过环境变量注入，不写入源代码。管理员通过受控的一次性命令创建。

## 16. 已确认的范围取舍

- 不对接真实支付；会计算和展示红包估值，并由管理员在前置流程完成后确认一笔带虚拟流水号的模拟打款。
- 不在演示服务器本地部署大模型。
- 不让 LLM 直接自由决定最终红包金额。
- 红包使用 20/25/25/15/15 的五维权重、二次金额曲线，高置信度重复投稿金额归零。
- 查重使用 pgvector 精确召回 + LLM 最终判定，Embedding 和生成模型均不锁定特定服务商。
- 生成模型使用 `json_object + 完整 Schema 提示词 + Pydantic/Python 后处理校验`；不把服务端 JSON mode 视为业务结构保证。
- 不为 MVP 用户会话引入 OAuth 或 Redis。
- 用户账号仅用于身份和数据隔离，不对接真实身份、银行卡、支付或提现账户；模拟打款不产生真实资金变动。
