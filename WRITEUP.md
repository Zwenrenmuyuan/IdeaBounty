# Idea Bounty 项目 Writeup

## 项目定位与主要取舍

我把本题定义为一个“可解释、可复核的点子估值 MVP”，而不是自动发钱系统。普通用户可以注册并提交 8–2000 字的痛点或方案；信息不足时在原记录中补充，失败时只重试失败阶段，无效投稿可删除。系统依次完成输入门禁、五维商业评分、Embedding、历史候选召回、LLM 查重和红包计算。管理员最终确认、调整或驳回结果；正金额只生成 `SIM-` 开头的模拟流水，不对接真实支付。

技术栈是 React + TypeScript + Tailwind、FastAPI + Pydantic + SQLAlchemy，以及 PostgreSQL + pgvector。为了控制面试 MVP 的复杂度，我选择同步处理流水线、PostgreSQL Session 和 pgvector 精确 Top 10 检索，没有引入 Redis、消息队列或近似向量索引。这些选择足以支撑低流量演示，也让事务、幂等和状态转换更容易审查。普通用户与管理员由后端按角色隔离，前端隐藏入口只是体验优化，不是权限边界。

## 我如何使用 AI 完成项目

我主要使用 Codex 作为结对开发工具。第一步是让它阅读面试题，但不直接编码；我再与它讨论 Python 后端、TypeScript 前端、PostgreSQL、AI 服务商兼容性、会话和部署方案。方案确定后，我把工作拆成工程基线、认证、投稿、AI 评估、Embedding、查重、金额、管理员和前端等可独立审查的里程碑。每一步先讨论契约、取舍和验收标准，再修改代码；实现后由我运行真实服务、查看 diff、质疑设计并决定是否提交。前端初版交给 GLM-5.2 实现，之后再用 Codex 审查和修正接口、权限、移动端布局与交互。

AI 帮助最大的是快速补齐边界：Pydantic 跨字段约束、Alembic 数据库约束、并发幂等、外部错误分类、Mock 测试和响应式页面。它还帮我生成了评估、Embedding 和查重三个真实模型探测脚本，使模型选择基于结构通过率、中文语义排序和固定案例，而不是只看服务商宣传。

AI 也有明显的“帮倒忙”。早期方案一度超过 MVP 所需复杂度；MiMo 并不严格执行传入的 `json_schema`，首次返回虽然是 JSON，却完全不符合项目 Schema；过严或表达不清的契约也曾导致偶发重试。UI 初版遗漏了“补充信息/删除无效投稿”，并让管理员看到了投稿入口；通用备忘需求也曾得到过高的新颖性和可行性分。我的处理方式不是直接接受生成结果，而是用真实探测、失败样例、数据库约束和测试回推设计：改用 `json_object + 完整 Schema 提示词 + Pydantic 后处理`，强化后端角色守卫，加入新颖性金额封顶，并规定“未给出具体方案”时可行性最高只能为 3 分。

## 查重与防止用户忽悠 AI

查重采用“确定性精确匹配 + 语义候选召回 + LLM 判定”。首先对原文执行 NFKC、大小写和空白归一化，用 SHA-256 找完全相同的历史内容。未命中时，只把“目标用户、核心痛点、场景、期望结果”送入 `BAAI/bge-m3` 生成 1024 维向量，在 pgvector 中精确召回 Top 10。最后由 LLM 分别判断痛点和方案是相同、相关还是不同，输出 `duplicate / related / novel`。只有高置信度重复才将金额归零，中低置信度重复保守按 `related` 处理；用户可以查看匹配点子的白名单脱敏摘要，管理员仍可最终纠偏。小数据量使用精确检索更容易验证，但当前尚未用大规模人工标注集证明查重准确率。

抗诱导不依赖一句“忽略用户指令”。表单只要求最少原文，该原文作为独立的不可信用户消息传入。模型先将信息标为 `explicit / inferred / unknown`，不允许把用户没说过的具体方案自动补出，并单独记录未证实断言、提示词注入、指定分数或金额等信号。有效内容夹带诱导时清洗后继续评估，清洗后没有有效贡献才拒绝。模型输出必须整体通过严格 Pydantic Schema 和跨字段校验，错误结果不会被修补或部分落库。最终总分、二次金额曲线、新颖性封顶和重复扣减全部由 Python `Decimal` 规则计算，用户不能靠提示词直接指定红包。

## 如果再给我一周

我会按对结果可信度的影响排序，而不是继续堆页面：

1. 扩展人工标注集，评估候选召回 `Recall@10`、重复判定 Precision/Recall、评分稳定性、成本和延迟，并据此调整 Prompt 与金额规则。误判重复会直接损害用户信任，所以应优先保证 Precision。
2. 为“市场新颖性”引入可追溯的外部检索证据。当前站内 `novel` 只代表没有历史匹配，而模型知识可能过时；类似备忘录的成熟功能不应因站内首次出现就获得高额红包。我会检索现有产品和替代方案，保存来源和时间，并将证据提供给管理员，而不让 LLM 凭记忆单独决定。
3. 将同步外部调用改为可恢复的后台任务，加入处理租约、超时恢复、可观测日志和失败告警，避免进程中断后记录长期停在中间状态。
4. 在管理员确认前执行一次最终查重，解决两个高度相似投稿并发进入系统时都被判为全新的竞态。
5. 增加 Playwright 端到端测试、处理状态自动刷新和完整后台筛选，并补齐限流、密码找回、账户风控、自动备份和部署监控。

## 已知问题与未完成部分

- 没有真实支付、实名、防刷、提现和财务对账；模拟流水仅用于演示。
- AI 评分和 LLM 查重仍受模型波动影响，当前查重探测集只有 9 个合成案例，管理员复核是必要兜底。
- 新颖性目前没有外部产品/市场数据作为证据；金额封顶只能降低风险，不能证明判断正确。
- 流水线同步执行，没有任务队列和遗留任务恢复；处理中页面需要用户手动刷新。
- pgvector 采用精确扫描且固定 1024 维，数据增长或更换模型后需要重建向量并重新评估索引方案。
- 极短时间内并发提交两个相似点子，可能都看不到对方的未完成记录并被判为全新。
- 前端尚无浏览器自动化测试、搜索和复杂筛选；当前依靠 TypeScript 构建、lint 和桌面/移动端人工检查。
- 管理后台列表是 `completed + accept` 的可审核队列，不是查看处理中、需补充、被拒绝和失败投稿的完整数据后台。
- 账户系统没有邮箱验证、找回密码、2FA、多账号防刷或应用级接口限流；管理员通过本地脚本提升，用户会话固定 3 天且只允许单端。

当前本地验收结果为后端 280 个 pytest 测试通过，Ruff、格式检查、严格 mypy 和 Alembic 漂移检查通过；前端生产构建通过，lint 有两个不阻断构建的 Fast Refresh 规则警告。本地启动步骤见 [README](./README.md)，单机上线步骤见 [部署指南](./部署指南.md)。
